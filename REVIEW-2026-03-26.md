# SDLC Workflow Review - 2026-03-26

## Scope

Review target:

- `sdlc-workflow/`
- `HANDOFF.md`
- `HANDOVER.md`
- `MEMORY.md`
- `plan-7.md`
- sample project: `/Volumes/HS-SSD-1TB/work-test-piple-1`

Review goal:

- determine whether execution drift came from model capability or workflow/skill design
- identify concrete defects in the skill source
- record the adjustments applied during this review

## Evidence Standard

This document distinguishes two evidence classes:

- **Verified**: confirmed by reading repository files or sample project files
- **Claimed**: stated in handoff documents but not supported, or contradicted, by verified artifacts

## Findings

### F1. Skill installation path was internally inconsistent

Severity: High
Evidence: Verified

- `HANDOVER.md` and `MEMORY.md` stated the skill should be linked into `~/.claude/skills/`
- `HANDOFF.md`, `sdlc-workflow/SKILL.md`, `sdlc-workflow/README.md`, and `sdlc-workflow/references/pipeline-overview.md` used `~/.agents/skills/`

Impact:

- operators can install the skill into the wrong location
- initialization instructions become ambiguous
- failures can be misattributed to model behavior when the entrypoint is misconfigured

Conclusion:

- this is a workflow/documentation integrity problem, not a model-quality problem

### F2. The review gates were documented as mandatory, but the source allowed silent downgrade

Severity: Critical
Evidence: Verified

Observed contradiction:

- `HANDOFF.md` declared dual-model review gates were mandatory and non-removable
- `sdlc-workflow/references/design-reviewer.md` and `sdlc-workflow/references/code-reviewer.md` originally said that if Codex CLI was unavailable, the pipeline could skip review

Impact:

- the most important control in the workflow could disappear at runtime
- downstream drift would appear as "model deviation" while the real failure was gate removal

Conclusion:

- this is a design defect in the skill

### F3. Chinese requirements could produce invalid or empty iteration slugs

Severity: High
Evidence: Verified

Original issue:

- `sdlc-workflow/references/requirements-ingestion.md` generated slugs by stripping non-ASCII characters
- for Chinese requirements, this can collapse to an empty slug or a low-quality identifier

Impact:

- invalid or colliding iteration directories
- poor traceability for Chinese-language operation, which is a primary target scenario

Conclusion:

- the workflow did not robustly support its expected language input

### F4. TG username auto-detection was promised more strongly than it was implemented

Severity: High
Evidence: Verified

Original mismatch:

- docs claimed `OPENCLAW_TRIGGER_USER` would be auto-detected and written to `.env`
- `sdlc-workflow/scripts/init-project.sh` did not actually create/update `.env`

Impact:

- TG/OpenClaw execution path could still stop on missing configuration
- operators would think runtime personalization existed when it did not

Conclusion:

- this was a workflow implementation gap

### F5. The test pipeline templates contained non-runnable shell logic

Severity: High
Evidence: Verified

Original defects in `sdlc-workflow/references/test-pipeline.md`:

- `UNIT_FAILED` and `E2E_FAILED` were read before guaranteed initialization under `set -u`
- helper functions like `run_unit_tests` and `run_e2e_tests` were referenced before being defined in the command template
- framework invocation patterns were inconsistent
- parallel execution guidance was optimistic and unsafe by default

Impact:

- the documented test stage could fail for scripting reasons unrelated to product code
- confidence in "test passed" claims is reduced

Conclusion:

- this is skill-spec quality debt, not a model hallucination

### F6. One command template referenced a nonexistent MCP call

Severity: Medium
Evidence: Verified

Original issue:

- `sdlc-workflow/references/requirements-ingestion.md` used `mcp__chrome_devtools__extract`
- no such tool exists in the current environment/tooling model

Impact:

- URL ingestion guidance could not be executed as written

Conclusion:

- the workflow mixed aspirational behavior with non-existent interfaces

### F7. The sample project did not support the claimed level of validation

Severity: High
Evidence: Verified

Observed sample-project inconsistencies:

- `tests/reports/h5-calculator-coverage.md` referenced `tests/unit/h5-calculator.test.ts`, but that file does not exist
- the sample project `package.json` contains a `lint` script, but no `eslint` dependency was present
- `HANDOVER.md` claimed the current workspace was not a Git repo, but the workspace actually contains `.git`

Impact:

- handoff confidence is lower than stated
- at least part of the reported test/verification story was not grounded in file-level checks

Conclusion:

- the sample project and handoff files were not being checked by a strict evidence loop

## Root Cause Assessment

Primary root cause:

- **skill design and workflow enforcement defects**

Secondary root cause:

- **handoff discipline was weak; claimed outcomes were not always reconciled against the real filesystem**

Not the primary root cause:

- raw model capability

Reasoning:

- most serious issues came from contradictory rules, missing guardrails, weak runtime enforcement, or unverifiable reporting
- these defects would create drift even with stronger models

## Residual Risks

The skill still has material open issues:

1. Installation-path language is still split across the handoff documents outside `sdlc-workflow/`
2. The sample project remains an unreliable proof fixture until its report/test/dependency story is repaired
3. The orchestrator is still largely specification-driven; there is not yet a fully executable end-to-end harness for the complete pipeline
4. Some reference files are examples rather than operational scripts, so future operators can still over-trust them

   - TG/OpenClaw initialization
   - gate failure behavior
   - unit/e2e/lint command presence
4. Add one operator-facing "source of truth" document for installation and execution paths
